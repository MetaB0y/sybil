# DoS / free-resource audit of the permissionless Sybil surface

Date: 2026-07-11  
Method: static code inspection only; no build, test, sequencer, deploy, or version-control commands were run.

Post-audit status: this remains the July 11 static evidence baseline, with a
living remediation ledger. Commit `a290e006e1aa` remediated findings 1, 3, 4,
and 6; `aec07c585d05` / GitHub #74 bounded limited-devnet anonymous onboarding;
and `b3b5a60a113a` / GitHub #86 replaced the bridge's single-RPC trust path with
an authenticated unanimous-finalized provider prefix. Finding 5 was later
re-based on the extracted `sybil-history` architecture. A remediation note does
not assert that any particular running deployment has loaded the cited change.

## Executive summary

The assumption that all state occupancy is bounded by deposited L1 capital
still does **not** hold. Anonymous devnet accounts are now finite and receive a
server-selected grant, but that grant is play money; cumulative private product
history remains an unbounded product policy, a projector outage can grow the
sequencer source outbox without a hard budget, and quarantine occupancy is
backed by only one token base unit per key.

At audit time, the most urgent issue was not an order: a zero-capital user
could grow an account's read-API-key vector without a count or string-size
bound and eventually cross qMDB's 1 MiB value limit. Commit `a290e006e1aa`
closed that wedge with a 64-record lifetime cap (revoked tombstones count), a
128-byte UTF-8 label cap, and a pre-mutation 256 KiB MessagePack budget for the
complete recovery account.

At audit time public onboarding created unlimited permanent accounts and let
the caller choose up to $5,000 of free balance. It now accepts only an initial
key, applies route/client flow limits, uses one configured grant, and refuses
the lifetime allocation after a default 1,000 account ids. Production posture
sets that grant to zero. Every allocated account still recurs in later
full-state witnesses, so the limit is a stock bound, not proof of capital-backed
admission; the real-value design is tracked in GitHub #89.

Ranked conclusions:

## Constants and quantitative baseline

### Account snapshot cost

An empty account snapshot encodes 96 bytes of fields: id (8), balance (8), total deposited (8), position count (8), events digest (32), and keys digest (32), plus the 21-byte `sybil/witness/account` domain, for **117 bytes** (`crates/sybil-verifier/src/snapshot_schema.rs:28-35`, `:169-171`). The full witness contains `pre_state`, `post_system_state`, and `post_state`, so a stable account costs **351 bytes/block** before vector-count overhead (`crates/sybil-verifier/src/witness_schema.rs:1023-1026`). Public creation requires an initial key; its account-key-set entry adds 8-byte account id, 8-byte key count, and a 38-byte key record, or **54 bytes** (`crates/sybil-verifier/src/witness_schema.rs:1040-1051`; key fields at `crates/sybil-verifier/src/types.rs:207-228`). Thus a normal empty public account adds roughly **405 canonical witness bytes every block**.

The typed state also gets an `acct/<u64>` leaf (`crates/sybil-verifier/src/state_schema.rs:22-26`, `:99-104`). Its application key/value material is approximately 125 bytes before qMDB structure overhead. There is **no** `acct_resv/` leaf for an empty account: reservation snapshots are emitted only for nonzero aggregate balance or position reservations (`crates/matching-sequencer/src/order_book.rs:1077-1118`).

### Order minimums and reservation cost

The exact numeric constants are:

- `NANOS_PER_DOLLAR = 1_000_000_000` (`crates/matching-engine/src/types.rs:10-11`).
- `SHARE_SCALE = 1_000`; `Qty(1)` is 0.001 share (`crates/matching-engine/src/types.rs:13-15`).
- `MAX_ORDER_QTY = 1_000_000 * SHARE_SCALE = 1_000_000_000` quantity units (`crates/matching-engine/src/types.rs:17-22`).
- Default maximum open resting orders: 1,000/account (`crates/sybil-api/src/config.rs:88-94`, `:306`).
- Default order TTL: 63,072,000 blocks (`crates/sybil-api/src/config.rs:48-50`, `:299`).

**No minimum quantity, minimum price, minimum notional, or positive tick was found.** Shape validation rejects only price above $1 and quantity above `MAX_ORDER_QTY`; it does not reject zero (`crates/matching-engine/src/order.rs:179-216`). Prices are integer nanos, so the smallest positive price representation is 1 nano, but zero is accepted.

For a pure buy, the exact reservation is:

```text
ceil(limit_price_nanos * quantity / 1_000)
```

(`crates/matching-sequencer/src/validation.rs:17-26`, `:45-54`). Therefore:

- `quantity = 0` reserves 0 at any valid price.
- `limit_price_nanos = 0` reserves 0 at any positive valid quantity.
- If both price and quantity are positive, the minimum reservation is exactly **1 nanodollar**.

A binary order's canonical order encoding is 34 bytes (`crates/sybil-verifier/src/canonical.rs:1-31`). A stable zero-reservation resting order appears as an accepted order event every block (about 69 bytes; carried orders are appended at `crates/matching-sequencer/src/sequencer/production/mod.rs:462-474`, encoding at `crates/sybil-verifier/src/event_schema.rs:247-253`) and in both pre- and post-sidecars (74 bytes each from `crates/sybil-verifier/src/snapshot_schema.rs:218-226`). It therefore adds roughly **217 canonical witness bytes/block/order**, excluding vector counters and solver allocations.

With the smallest positive reservation, the capital-only theoretical density is **1 billion orders/$1**, or about **217 GB of recurring canonical witness/$1** before account and reservation overhead. The per-account 1,000-order cap means realizing that density needs one million accounts per dollar, but account creation is currently free. At zero price the ratio is unbounded because the denominator is $0.

### Flow controls are not stock controls

Order/cancel defaults are 50 submissions/account/second (burst 200), 1,000 global/second (burst 3,000), and 1,000 open orders/account (`crates/sybil-api/src/config.rs:60-102`, `:302-307`). A second HTTP-edge order limiter defaults to 500 global requests/second and 250/client/second (`crates/sybil-api/src/config.rs:104-119`, `:308-311`). These apply only to the three order paths (`crates/sybil-api/src/app.rs:434-458`). Account, signing-key, profile, and API-key paths do not pass through this limiter.

The sequencer enforces the per-account open-order cap on the write path, counting live and staged non-MM orders (`crates/matching-sequencer/src/actor/handlers/admission.rs:251-268`). It also enforces the global submission bucket before orders and cancels (`crates/matching-sequencer/src/actor/handlers/dispatch.rs:126-174`). These are useful flow controls, but neither caps the number of accounts nor total open orders across all accounts.

## Ranked findings

### 1. CRITICAL — public read-API-key growth could permanently wedge block persistence — REMEDIATED

**Post-audit remediation (`a290e006e1aa`).** Admission now counts revoked
tombstones toward `MAX_API_KEYS_PER_ACCOUNT = 64`, limits labels to
`MAX_API_KEY_LABEL_BYTES = 128` UTF-8 bytes, serializes the candidate recovery
account before nonce/WAL mutation, and rejects it above
`MAX_SERIALIZED_ACCOUNT_BYTES = 262,144` (256 KiB). qMDB's codec limit remains
1,048,576 bytes, leaving 75% headroom. The evidence below describes the code
as inspected before those bounds landed.

**Exact action.** Create one free account and repeatedly call `POST /v1/accounts/{id}/api-keys` with valid signed nonces. Either use one near-request-limit label or many smaller labels/keys. Revoking keys does not repair the account.

**Code path.** The route is public (`crates/sybil-api/src/app.rs:520-527`, `:846-854`). The handler accepts `req.label` without any length validation and forwards it into the signed mutation (`crates/sybil-api/src/routes/accounts.rs:1093-1101`, `:1140-1164`). No count check exists before `account.api_keys.push(...)` (`crates/matching-sequencer/src/sequencer/accounts.rs:305-334`). Revocation merely sets `revoked_at_ms` and removes the hash from the lookup index; the record remains in the vector (`crates/matching-sequencer/src/sequencer/accounts.rs:337-359`; retention is also documented in `crates/matching-sequencer/src/account.rs:61-64`).

Every block persists the full `Account`, including `profile` and `api_keys`, into the account-recovery qMDB (`crates/matching-sequencer/src/account_storage.rs:126-144`; serialization at `crates/matching-sequencer/src/qmdb_accounts.rs:210-223`). That qMDB codec is configured with `MAX_VALUE_BYTES = 1 << 20` (`crates/matching-sequencer/src/qmdb_accounts.rs:25-32`, `:190-199`). Persistence happens before the redb fence is committed (`crates/matching-sequencer/src/store/commit.rs:699-726`). A value over the codec range makes the block persist path fail; the actor then retries rather than advancing (`crates/matching-sequencer/src/actor/production.rs:147-154`).

**Capital/authentication.** No capital. A signature from the attacker's own free account is required. There is no route rate limit, key count cap, aggregate byte cap, or deletion operation. The normal JSON extractor imposes only the framework request-body ceiling; cumulative small records bypass any per-request ceiling.

**Resource and permanence.** Sequencer heap (`Vec<ApiKeyRecord>` plus strings), control-plane WAL, qMDB account snapshots, serialization/copy CPU, and finally block liveness. The mutation is recovery-durable because it is WAL-appended before being applied (`crates/matching-sequencer/src/actor/handlers/control_plane.rs:463-483`; `crates/matching-sequencer/src/store/wal.rs:147-153`). Revocation cannot shrink it, and no account/API-key deletion path was found. API keys are not in `AccountSnapshot`, so this does **not** directly enlarge the validity witness or `acct/` leaf; it wedges the parallel full-account recovery snapshot.

**Bound.** None in product logic. The 1 MiB qMDB value ceiling is a failure threshold, not a safe admission bound.

### 2. CRITICAL — unlimited public accounts minted free balances and entered every future witness — REMEDIATED FOR LIMITED DEVNET

**Post-audit remediation (`aec07c585d05`, GitHub #74).** Anonymous creation is
now `POST /v1/onboarding/accounts` with a deny-unknown-fields, initial-key-only
DTO. The server chooses one configured grant, dedicated global/client token
buckets bound request flow, and the bootstrap lock atomically refuses account
allocation once the durable lifetime `next_account_id` stock reaches
`SYBIL_PUBLIC_ACCOUNT_CAPACITY` (default 1,000). IDs are not reclaimed or
reused. `POST /v1/accounts` is service/dev-only and explicitly owns operator
funding. Production configuration fixes the anonymous grant at zero and
preflight rejects a nonzero override. This is sufficient for the scoped public
devnet; it is not a capital-backed real-value admission design, which is
tracked in GitHub #89. The evidence below describes the pre-remediation route.

**Exact action.** Repeatedly call `POST /v1/accounts` with a fresh syntactically valid compressed P256 key and any `initial_balance_nanos` from 0 through 5,000,000,000,000. Possession of the raw-P256 initial key is not proved by account creation; parsing only checks that the point is valid (`crates/sybil-api/src/routes/accounts.rs:93-106`, `:237-264`).

**Code path.** The route is public (`crates/sybil-api/src/app.rs:500-503`, `:810-818`). Anonymous callers are capped only at **$5,000 of newly created in-protocol balance per account**, not required to deposit (`crates/sybil-api/src/routes/accounts.rs:36-42`, `:67-92`). After a duplicate-key precheck it calls `create_account`, then registers the first key (`crates/sybil-api/src/routes/accounts.rs:108-139`). `AccountStore::create_account` monotonically allocates and inserts a permanent account (`crates/matching-sequencer/src/account.rs:122-142`); no account removal/closure path was found. Creation also stages a `CreateAccount` system event and a durable history event (`crates/matching-sequencer/src/sequencer/accounts.rs:438-461`).

Witness phase construction includes every untouched account (`crates/matching-sequencer/src/sequencer/production/witness.rs:83-103`), and witness assembly stores all three account states plus all active key sets (`crates/matching-sequencer/src/sequencer/production/witness.rs:363-385`, `:394-410`). The typed state creates one `acct/` leaf/account (`crates/sybil-verifier/src/state_schema.rs:22-26`, `:99-104`). State qMDB persistence rebuilds and merkleizes the complete leaf set into a new generation every block (`crates/matching-sequencer/src/qmdb_state.rs:463-514`).

**Capital/authentication.** Zero L1 capital and no existing identity. The supplied key must be a valid unused P256 point. No IP, per-key, global creation-rate, proof-of-work, deposit, reservation, account-count, or tombstone-reclamation bound was found. The bootstrap mutex only serializes concurrent creations (`crates/sybil-api/src/routes/accounts.rs:108`); it is not backpressure or a rate cap. The HTTP order limiter does not cover this path.

**Resource and permanence.** Permanent account map entry, recovery qMDB entry, typed `acct/` leaf, active pubkey registry entry, one durable creation-history row, and about **405 recurring witness bytes/account/block**. Continuous creation also makes retained DA history superlinear in time: each later full snapshot contains all earlier accounts. Full-block/DA history pruning defaults to disabled (`canonical_archive_retention_blocks = 0`) at `crates/sybil-api/src/config.rs:315-319`; zero means no pruning (`crates/matching-sequencer/src/store/retention.rs:38-66`).

**Bound.** Only u64 account-id exhaustion and machine resources. `MAX_PUBLIC_DEMO_BALANCE_NANOS` limits free balance **per account**, not account count or state.

### 3. HIGH — resting orders could reserve exactly zero capital — REMEDIATED

**Post-audit remediation (`a290e006e1aa`).** Non-MM orders that may enter the
durable resting book must now have `quantity > 0`, `price > 0`, and ceiling
notional at least `SYBIL_MIN_RESTING_ORDER_NOTIONAL_NANOS`. The default is
1,000,000 nanodollars ($0.001). One-shot MM quotes do not rest and retain their
existing shape rules. The historical analysis below remains useful for the
pre-remediation path; a finite play-money grant still is not L1 capital, but
account stock is now bounded under finding 2.

**Exact action.** On each free account with a positive free balance (1 nano suffices to survive revalidation), submit up to 1,000 signed GTC buys with `limit_price_nanos = 0` and any valid positive quantity. Alternatively submit `quantity = 0`. Avoid a crossing counterorder so the orders rest.

**Code path.** Zero price and zero quantity pass `validate_binary_one_hot` (`crates/matching-engine/src/order.rs:179-216`). Buy reservation uses ceiling notional and returns zero for either case (`crates/matching-sequencer/src/validation.rs:17-26`, `:45-54`). `OrderBook::accept` stores the order even when `cost == 0`; it merely skips the aggregate balance-reservation increment (`crates/matching-sequencer/src/order_book.rs:386-433`). An `order/` leaf is committed for every resting order (`crates/sybil-verifier/src/state_schema.rs:76-83`, `:127-131`). Since zero aggregates are omitted, these orders do not create `acct_resv/` leaves (`crates/matching-sequencer/src/order_book.rs:1077-1118`).

**Capital/authentication.** No L1 capital. Signed account ownership is required, but the account and its positive balance are free under finding 2. Even if free balance minting were removed, price-zero orders would still have zero reservation; only the current `account.balance <= 0` revalidation eviction (`crates/matching-sequencer/src/order_book.rs:511-518`) forces the account to retain some positive balance.

**Resource and permanence.** `order/` leaves, order-book heap/persistence, full pre/post sidecars, accepted-order witness events every block, solver input, state-root rebuild, proving, and DA. Approximate recurring canonical witness cost is **217 bytes/order/block**. At 1,000 orders/account that is about 217 KB/account/block in orders alone, plus the account.

**Bound.** 1,000/account is correctly enforced, order flow is rate-limited, and TTL is 63,072,000 blocks. There is no total-order cap, nonzero-reservation requirement, or capital-per-leaf floor. Unlimited free accounts make total stock unbounded at $0.

### 4. HIGH — free signing-key churn could exceed the validity limit — REMEDIATED

**Post-audit remediation (`a290e006e1aa`).** Authorized registration and
revocation now count pending `KeyRegistered` and `KeyRevoked` events and reject
the 65th operation before WAL mutation. The admission constant is the
verifier-owned `MAX_KEY_OPS_PER_BLOCK = 64`, preventing the sequencer/verifier
split described below. This does not prune already committed key-operation
history; that residual storage concern belongs to open finding 5.

**Exact action.** On an owned free account, alternate signed key registration and revocation between block ticks. Keep at least one key and below the active-set cap, but submit more than 64 successful operations before the next block.

**Code path.** Public register/revoke routes are mounted without the order-rate middleware (`crates/sybil-api/src/app.rs:508-515`, `:834-841`; limiter scope at `:434-458`). Active registration is capped at `MAX_KEYS_PER_ACCOUNT = 16` (`crates/sybil-verifier/src/account_keys.rs:7-11`; sequencer enforcement at `crates/matching-sequencer/src/sequencer/accounts.rs:51-75`), and last-key revocation is blocked (`crates/matching-sequencer/src/sequencer/accounts.rs:221-241`). Successful operations are immediately pushed into `pending_system_events` and update the account digest (`crates/matching-sequencer/src/sequencer/accounts.rs:133-160`, `:187-218`).

The verifier independently rejects a witness with more than `MAX_KEY_OPS_PER_BLOCK = 64` key events (`crates/sybil-verifier/src/key_transition.rs:28-43`). No corresponding pending-key-op admission check was found in the API, actor handlers, or sequencer. Production explicitly sets inline full verification off (`crates/sybil-api/src/main.rs:89-90`), so the sequencer can commit/serve a witness that the validity verifier/prover rejects rather than stopping it at admission.

**Capital/authentication.** No capital. Each operation requires a current account-key signature and fresh digest binding. No per-account, IP, HTTP, or global key-op rate exists.

**Resource and permanence.** P256/WebAuthn verification CPU, WAL rows until block commit, potentially unprovable block, and retained block/DA event history. Raw-P256 key events are roughly 163 canonical bytes each; WebAuthn authorizations can be much larger, although verifier decoding caps authenticator data at 512 bytes and client JSON at 2 KiB (`crates/sybil-verifier/src/account_keys.rs:9-11`, `crates/sybil-verifier/src/witness_schema.rs:619-636`). Active recurring witness state is locally bounded to 16 key records/account; revocation removes the active registry row, so churn does not leave active-key tombstones.

**Bound.** Active set 16/account and verifier limit 64 ops/block. The latter is not enforced on the write path, and account count is unbounded.

**Post-audit metadata remediation.** Initial, first-key-bootstrap, and signed
additional-key registration now share a 128-byte UTF-8 signing-key-label cap.
The original bytes are neither trimmed nor normalized. Atomic onboarding
checks the cap before account allocation, while the actor and sequencer check
it again before WAL or registry mutation. The active-key cap still bounds this
metadata per account; limited-devnet account creation is now lifetime-bounded
under finding 2.

### 5. HIGH — cumulative product history and projector-outage backlog remain unbounded — OPEN, DEVNET OBSERVABLE

**Architecture correction.** The sequencer-local `HISTORY_EVENTS`,
`FILL_HISTORY`, `EQUITY_POINTS`, price, and candle query tables described by the
original audit no longer exist. A fenced block now writes one immutable private
`CommittedHistoryBatchV1` row to `history_outbox_v1` in the same redb
transaction as canonical commit. `sybil-api` delivers oldest prefixes at least
once, and `sybil-history` atomically stores the raw batch, all projections, and
the contiguous checkpoint. Only a verified checkpoint and exact payload hash
allow the source row to be deleted. Historical queries bypass the sequencer
actor and expose lag/completeness rather than returning false empty data.

**Exact action.** Repeatedly place and cancel valid minimum-notional orders, or
trade between funded accounts, while the history projector is unavailable,
misconfigured, slower than ingress, rejecting a permanent conflict, or unable
to write its volume. Zero-price resting orders are no longer valid and
anonymous account count is finite, but the same authorized accounts and
capital can generate new facts indefinitely.

**Capital/authentication.** Mutations require an account signature and obey
HTTP/actor flow limits. Devnet grants are play money; with real collateral the
same capital can still be churned repeatedly, so cumulative history volume is
not bounded by capital stock.

**Resource and permanence.** During normal operation acknowledged source rows
are short-lived while `sybil-history` retains raw private batches and serving
indexes under its own product/legal archive policy. During an outage, one
encoded batch per committed block accumulates on the sequencer filesystem that
also holds canonical state. Canonical block/DA retention is separate and cannot
be substituted for a missing private batch. Silent source deletion would break
history completeness; refusing a new row inside the commit fence would halt
block commits. GitHub #90 therefore requires an explicit overflow/archive ADR.

**Current bound and detection.** Per-request and per-block flow is layered, but
neither the source backlog nor the history archive has a ratified row/byte/age
budget. The source now transactionally maintains exact encoded payload bytes
and exports rows, bytes, oldest/newest height, and oldest age even without a
configured projector. Alerts fire for a 15-minute-old source row, a 256 MiB
logical backlog, and 15%/5% root-filesystem headroom. These are devnet incident
signals, not a hard bound or permission to drop rows.

### 6. HIGH — public DA manifest was an unrate-limited full-payload hashing oracle — REMEDIATED

**Post-audit remediation (`a290e006e1aa`).** Block commit now writes the small
`DaArtifactManifest` into a separate `da_manifests` row atomically with the
payload artifact. The public manifest path reads that metadata row without
loading, allocating, or hashing witness bytes. Both retained DA routes pass
through a dedicated limiter before store dispatch: defaults are 20 global
requests/second with burst 40, 10 per client/second with burst 20, and four
in-flight requests. The original trace below records the old handler.

**Exact action.** Repeatedly request `GET /v1/da/{height}/manifest` for retained large-witness heights, concurrently and/or across heights.

**Code path.** The manifest route is public (`crates/sybil-api/src/app.rs:496-499`, `:806-809`). To return the small manifest, the handler fetches the complete `DaArtifact` and calls `verify_da_artifact` (`crates/sybil-api/src/routes/da.rs:25-31`). Fetching is sent through the single sequencer actor, which loads/deserializes the full artifact from redb (`crates/matching-sequencer/src/actor/handlers/dispatch.rs:363-387`; `crates/matching-sequencer/src/store/da.rs:186-194`). Integrity verification hashes the entire payload (`crates/matching-sequencer/src/store/da.rs:104-122`). There is no general read-rate, concurrency, or timeout layer in `create_router`; only order POSTs are limited (`crates/sybil-api/src/app.rs:1191-1217`).

**Capital/authentication.** None; no token or signature.

**Resource and permanence.** Per-request redb read, full artifact allocation/deserialization, full-payload hash CPU, actor occupancy, and a small response. Amplification scales with the witness size created by findings 2 and 3. This is flow-only: the read does not grow persistent state.

**Bound.** Only retained-artifact size, server/network concurrency, and machine resources. There is no application rate limit. Default retention keeps all DA artifacts (`crates/matching-sequencer/src/store/da.rs:154-183`; retention default cited above).

The more direct expensive endpoints are correctly service-gated in production: state proofs and `GET /v1/da/{height}/payload` are in `SERVICE_ROUTE_TABLE` (`crates/sybil-api/src/app.rs:666-678`, handlers mounted at `:988-995`) and protected by fail-closed bearer middleware (`crates/sybil-api/src/app.rs:1156-1175`, `:1191-1194`). State proof generation itself performs live qMDB inclusion/exclusion proof work (`crates/matching-sequencer/src/actor/queries.rs:242-311`), but it is not permissionless when `dev_mode=false`. Payload transfer is likewise not permissionless. This is a **no public issue found** result for those two exact endpoints.

Public `/v1/blocks`, block-by-height, SSE, and WebSocket endpoints also have no general rate limit. Block list response count is capped at 500 (`crates/sybil-api/src/routes/blocks.rs:14-45`), but repeated historical reads and unlimited streaming connections remain bandwidth/actor flow risks. They do not serve the full witness and do not themselves grow state, so they rank below the manifest amplification.

Two other public reads become linear scans after state has already been inflated. `/metrics` performs joined live-market/bridge queries, clones bridge state, and folds the full quarantine ledger (`crates/sybil-api/src/app.rs:244-345`); `/v1/bridge/status` counts every withdrawal and folds quarantine (`crates/sybil-api/src/routes/bridge.rs:122-166`). The metrics collector has a two-second internal timeout, but neither route has caller rate limiting. These are secondary flow amplifiers rather than independent stock-growth vectors.

### 7. MEDIUM — quarantine occupancy is capital-backed, but only by one token base unit per key — OPEN

**Exact action.** On L1, call `SybilVault.deposit(1, unique_unresolvable_sybil_account_key)` repeatedly. The trusted indexer/service submits the sequential finalized deposits; each unique key becomes a quarantine ledger entry.

**Code path.** The vault rejects only `amount == 0`; it has no larger minimum (`contracts/src/SybilVault.sol:159-177`). Sequencer conversion likewise accepts every nonzero token-unit amount and multiplies by `NANOS_PER_TOKEN_UNIT = 1_000` (`crates/matching-sequencer/src/bridge.rs:12`, `:264-275`). A quarantined deposit inserts or aggregates by its 32-byte key in the `BTreeMap` and stages a system event (`crates/matching-sequencer/src/sequencer/bridge_ops.rs:171-220`). The full quarantine opening is copied into both witness sidecars; each entry encodes 32-byte key plus 8-byte amount (`crates/sybil-verifier/src/snapshot_schema.rs:294-313`). State root commits only a single quarantine digest leaf (`crates/sybil-verifier/src/state_schema.rs:36-42`), but proving/DA receives the full ledger.

**Capital/authentication.** Yes: at least one real token base unit is transferred to the vault, plus L1 gas. For the configured 6-decimal dollar-token conversion, one unit is $0.000001 (1,000 nanodollars). The HTTP deposit ingestion endpoint is service-token gated (`crates/sybil-api/src/app.rs:691-698`, `:1004-1007`), so an internet caller cannot fabricate quarantine entries directly. Post-audit GitHub #86 replaced the old handler trust assumption: the indexer now requires at least two uniquely identified providers, chooses their lowest common finalized height, requires unanimous header hashes, binds log reads to the agreed block hash, and reconciles vault storage at that hash across every provider before service-authenticated submission. This is an explicit at-least-one-honest-independent-provider model, not an Ethereum light client; the deeper cryptographic path is GitHub #88.

**Resource and ratio.** About 40 bytes of canonical ledger opening per sidecar, or **80 recurring witness bytes/unique key/block**, excluding vector overhead and BTreeMap allocation. At one microdollar/key, $1 can create one million entries and approximately **80 MB of recurring witness**, plus roughly 40 MB of raw ledger material and allocator/tree overhead. L1 transaction gas makes the practical dollar cost much higher, but occupancy is weakly bounded by deposited token capital.

**Bound.** One entry per unique 32-byte key; repeated deposits to the same key aggregate. The deposit accumulator is a depth-32 tree, a protocol maximum of `2^32` deposits (`crates/sybil-l1-protocol/src/lib.rs:28`, capacity calculation at `:405`; vault depth at `contracts/src/SybilVault.sol:13`). No quarantine entry-count cap or minimum above one base unit exists.

## Checked surfaces with no permissionless stock issue found

### Account reservations and removed orders

- Empty accounts do not create `acct_resv/` leaves. Only accounts with nonzero aggregate reserved balance/positions do (`crates/matching-sequencer/src/order_book.rs:1077-1118`).
- Cancelled, fully filled, expired, or invalidated orders are removed from the live book and their reservations released (`crates/matching-sequencer/src/order_book.rs:435-472`, `:680-704`). No live `order/` tombstone remains. Append-only history remains as described in finding 5.

### Profiles

Public profile mutation is signed and nonce-protected. Display name is capped at 32 characters and avatar seed at 64 bytes (`crates/sybil-api/src/routes/accounts.rs:519-565`). Updates replace two optional strings rather than append (`crates/matching-sequencer/src/sequencer/accounts.rs:278-298`). No unbounded stock growth was found, although the route has no flow rate limit.

### Deposits credited to a real account

The cumulative historical deposit list was replaced by a fixed 32-hash frontier plus only this block's new deposits (`crates/sybil-verifier/src/types.rs:400-428`). Credited deposits change an existing account's fixed balance/deposit fields rather than leave one state leaf per deposit (`crates/matching-sequencer/src/sequencer/bridge_ops.rs:176-202`). Deposit facts still enter the private committed history batch and remote projection described in finding 5, but canonical recurring state per credited deposit does not grow.

### Withdrawals

All withdrawal routes, including the signed variant, are currently in the service route tier (`crates/sybil-api/src/app.rs:695-706`, `:1008-1019`), so a permissionless caller cannot create withdrawal leaves. Creation burns/deducts a nonzero amount and checks available unreserved balance (`crates/matching-sequencer/src/sequencer/bridge_ops.rs:293-325`, `:328-381`). Terminal withdrawals are removed in the block carrying finalization/refund (`crates/matching-sequencer/src/sequencer/production/mod.rs:363-375`). No permissionless stock issue was found on the mounted production surface.

### Market, group, feed, raw-event, reference-price, metadata, and resolution writes

These state-growing writes are all service-tier routes (`crates/sybil-api/src/app.rs:712-758`, `:1024-1068`) and receive the same fail-closed service bearer middleware. A permissionless user cannot create markets/groups or extend their lifetime/state. This is a **no public issue found** result; operator credentials remain capable of uncapitalized growth by design.

### Unsigned orders and direct funding

`POST /v1/orders` and `/v1/accounts/{id}/fund` are service-tier only (`crates/sybil-api/src/app.rs:666-685`, `:980-999`). Permissionless trading uses signed single-order submissions. The public account-creation balance grant in finding 2 nevertheless provides equivalent free capital without using `fund`.

## Remediation status and remaining priority

The original audit did not modify product code. Subsequent changes produced the
following current ledger:

1. **REMEDIATED:** read API keys are capped at 64 lifetime records including
   revoked tombstones, labels at 128 UTF-8 bytes, and the complete serialized
   recovery account at 256 KiB before mutation/WAL admission.
2. **REMEDIATED for limited devnet:** anonymous creation accepts only an initial
   key, uses a fixed server grant and dedicated flow limits, and has a durable
   lifetime account-stock ceiling. IDs are not reclaimed. Capital-backed
   real-value admission and state economics remain GitHub #89.
3. **REMEDIATED for the identified zero-reservation path:** durable non-MM
   resting orders require positive quantity, positive price, and a configurable
   minimum notional (default 1,000,000 nanodollars). A global live-order policy
   may still be warranted within the finite account stock.
4. **REMEDIATED for validity alignment:** the sequencer rejects the 65th pending
   key operation using the verifier's 64-op constant. Signing-key labels are
   also capped at 128 UTF-8 bytes before account/key/WAL mutation. A broader
   shared control-plane rate limit remains defense in depth, not the validity
   fix.
5. **OPEN, DEVNET OBSERVABLE:** the extracted private history service owns raw
   batches and serving projections; the sequencer owns only the unacknowledged
   transactional outbox. Exact source rows/payload bytes/heights/age plus host
   disk alerts are implemented. GitHub #90 must ratify source overflow/archive
   semantics and longer-term private-history retention.
6. **REMEDIATED for the manifest oracle:** serve cached manifest metadata and
   rate/concurrency-limit retained DA reads. Other public historical/streaming
   endpoints may still merit independent bandwidth and connection limits.
7. **OPEN:** set an L1 minimum deposit or charge a state-occupancy fee high
   enough that a quarantine key's lifetime witness cost is capitalized.

The universal economic-safety assumption remains false because limited-devnet
accounts are deliberately not proof-of-capital admission, cumulative history
can outgrow reused capital, and quarantine dust weakly capitalizes recurring
witness occupancy. The immediate public-devnet stock wedge from unlimited
anonymous accounts is closed; real-value economics remain #89, product-history
overflow remains #90, and quarantine economics remain item 7.
