Sybil is safer and cleaner than its line count suggests—but it has not completed several successful extractions.

Executive verdict

The repository did not become simpler in raw size; it became more explicit and safer. Recent work added real complexity for durability, proof retention, recovery, provider quorum, history isolation, and truthful failure states. Most of that complexity is justified. The simplification opportunity is now to remove the old path after the new path landed, and to stop research or optional products from sharing production ownership.

Delete now

The superseded prover mock binary, the no-op viz feature, the broken sim-eg recipe, and four source-unreferenced direct dependencies.

Isolate

Research solvers, native-market provisioning, Arena analytics, and optional tracing should not broaden the default exchange runtime or production compile surface.

Make product decisions

LLM auto-resolution, the public Arena, SSE, and reserved oracle states each have a cost across multiple layers. Keep them only with an owned milestone.

Keep deliberately

The actor single-writer, qMDB/redb fence, verifier/guest separation, transactional history outbox, proof protocol, and integer truth are good complexity.

Highest-leverage recommendation: finish the accepted history-service extraction. Public historical reads already use sybil-history, but the sequencer still retains several old in-process history query caches and their persistence/configuration surface. Completing that deletion makes the new service a simplification rather than an addition.

What the line count overstates

  • The generated Python OpenAPI client is about two thirds of tracked Python lines, but is mechanically regenerated and is not two thirds of the authored design burden.
  • Large JSON is mostly frozen market, benchmark, or simulation data. It affects clone/review weight, not runtime architecture.
  • The small protocol crates are dependency firebreaks. A lower crate count would not necessarily mean a simpler system.
  • Verifier schemas are intentionally repetitive at native, canonical, witness, and guest boundaries. Consolidate encoders only when byte identity remains explicit and testable.

Ranked proposals in detail

1. Complete the history-service extraction by deleting the retained query caches

Highest leverage

The system paid the cost of adding sybil-history, but still carries parts of the embedded predecessor. The accepted design explicitly says the extraction becomes simpler only after the old serving tables, query RPCs, pruning, and hot projections are removed.

Evidence: account fills/events/equity and market price/candle REST routes now use HistoryClient. Meanwhile AnalyticsState still owns FillRecorder, EquityTracker, AccountEventLog, and retained PriceTracker points; API config still exposes four max_recent_* knobs. Direct public-route consumers of the old query methods were not found.

Proposal. Keep current marks, total counters, cost basis, live aggregates, rolling 24-hour anchors/volume, and the pending outbox deltas. Remove the per-account recent fill rows, equity rows, event rows, general price-history rows, now-dead restore fields, query methods/messages, cache metrics, and configuration. Replace cache-shaped snapshot fallback plumbing with a direct committed-fact accumulator. Refactor PriceTracker into current/rolling state plus that accumulator so 24-hour stats are not accidentally deleted.

Net result: one owner for historical queries, less sequencer memory and restore/snapshot plumbing, fewer config knobs, and a clearer validity-versus-product-data boundary.

2. Compile only the production solver in the production feature

Keep research

Production always instantiates RetainedCashSolver, but matching-sequencer enables the broad lp feature. That same feature exports the LP-SLP baseline, pacing bundle, and decomposed solver. The capability is valuable; the ownership is too broad.

Evidence: RetainedCashSolver is selected in both fresh and restored sequencers. Under lp, raw module sizes are approximately 1,593 lines for LP-SLP, 563 for pacing, 1,464 for decomposed, and 1,039 for retained-cash. Conic and MILP are already optional. Recent benchmark reports explicitly keep pacing and Clarabel as research candidates.

Proposal. Rename/split the feature around intent: a retained-cash production feature containing HiGHS oracle/shared landing code and the production solver; a research-solvers feature or lab crate for LP-SLP, pacing, decomposed, Conic, MILP, experiments, and benchmarks. Do not delete pacing or decomposed while the solver and payoff-vector research roadmaps remain active.

Net result: the deployed exchange exposes one algorithm, while the research laboratory remains rich and independently testable.

3. Delete the old sybil-prover-mock polling binary

Safe removal

There are two mock proving paths. The restart-safe prover daemon has a typed mock backend and is the path Compose and soak tests use. The feature-gated polling mock writes the older per-block artifact flow.

Evidence: mock_live.rs plus its bin wrapper are 356 raw lines and pull optional sybil-api-types. No recipe, compose service, runbook command, or deployment invokes sybil-prover-mock. Base Compose explicitly runs sybil-prover daemon with SYBIL_PROVER_PROOF_KIND=mock.

Proposal. Remove the mock-live feature, optional API-types dependency, bin target, module, and old producer naming where it is only legacy compatibility. Retain the daemon's typed mock backend for cheap integration.

Net result: one durable proof ingestion/acknowledgement model in all environments.

4. Make OpenTelemetry complete or absent

No half-state

The API includes an OTLP exporter, but the repository has no deployed collector/backend, no end-to-end correlation, and almost no useful span tree. This is a documented “half-installed seed,” not an operational tracing system.

Evidence: four direct OTel dependencies are unconditional in sybil-api; export occurs only when OTEL_EXPORTER_OTLP_ENDPOINT is set; no compose profile sets it. Commonware pulls OTel 0.31 while the API pulls 0.32, so the API graph contains both stacks. The design note is marked proposal-needs-revalidation.

Proposal. Choose one of two coherent states. If causal tracing is a funded near-term task, gate it behind an otel build feature and land the collector/Tempo profile, correlation context, block spans, prover spans, and sampling policy together. If not, remove the API exporter/init dependencies until that project starts. Keep Prometheus, structured tracing logs, and alerts either way.

Net result: no compiled telemetry stack that produces no usable traces.

5. Remove stale solver visualization switches; repair or archive the dashboard

Safe core cleanup

The viz feature claims to enable detailed phase snapshots but has no conditional code. The Streamlit loader still expects fields that the current exporter never emits. A just recipe still uses a solver name removed by an earlier cleanup.

Evidence: no cfg(feature = "viz") exists; VizSnapshot emits final result, timing, orders, and fills only; dashboard pages expect phase_snapshots and liquidity. just sim-eg exits 2 because eg is not a valid solver choice.

Proposal. Delete the empty feature and sim-eg. Move the JSON snapshot type from matching-solver to matching-sim, where export ownership belongs. Then either update the roughly 2.1k-line Streamlit dashboard to the actual schema and add a smoke test, or archive/delete it and keep the CLI/benchmark reports.

Net result: commands and feature flags tell the truth; simulation presentation no longer leaks into the solver library.

6. Remove the API's direct dependency on Arena's SQLite schema

Boundary repair

The architecture says Python Arena connects to Rust over HTTP, but the main API also opens the Python-owned decisions.db volume and knows its schema. This reverses ownership and makes a bundled SQLite library part of the exchange server.

Evidence: sybil-api has bundled rusqlite, an arena_db_path config/state field, 118 lines of Arena metrics access, and a 1,038-line bots route module. Compose shares arena-data. The public web Arena view is 2,008 TSX lines and also feeds the activity page.

Proposal. Make a product choice. If Arena is a first-class public surface, let Arena expose a small typed read API or publish stable facts into a history-owned projection; sybil-api may proxy/authorize but must not read Python storage. If Arena is an experiment, remove /v1/bots/* and /arena from the main exchange and keep the separate Arena dashboard.

Net result: no cross-language database schema coupling, no SQLite C dependency in the API, and an honest product boundary.

7. Stop making the Polymarket mirror own native provisioning and optional LLM resolution

Product boundary

sybil-polymarket currently mirrors Polymarket, creates native catalog markets/groups, bootstraps their MM, polls arbitrary resolution sources, calls an LLM, signs proposals, and interacts with an API review board. Those are three products with different trust and failure models.

Evidence: native provisioning is active through Compose; LLM auto-resolution is default-off and not enabled by current compose overlays. The native, auto-resolve, and LLM modules alone are 1,992 raw lines, before sync/mapping/API/store glue. The resolver review surface adds four service routes and sequencer persistence, but core settlement still uses immediate signed attestations.

Proposal. Keep native markets if they are part of the product. Move catalog application to an idempotent one-shot admin/import command rather than a Polymarket actor. Put the resolver/review queue behind its own explicit experimental package or delete it until an enabled deployment and owner exist. The Polymarket sidecar should mirror, quote, and attest Polymarket events only.

Net result: integrations are named by what they own, and an optional LLM can fail without broadening the mirror or core API.

8. Standardize first-party block consumption on resumable WebSocket

Unify transport

The API maintains REST replay, public SSE, public resumable WS, and service-authenticated canonical WS. Public WS is the documented recovery protocol; first-party Python still uses SSE without replay despite its README saying “WebSocket-first.”

Evidence: Rust client and WS integration tests use /v2/blocks/ws?from_block=N. Python stream_blocks() connects only to /v1/blocks/stream and reconnects without a height cursor. Arena bots, runner, and live analyst consume that method.

Proposal. Implement versioned public WS/replay in the Python wrapper, migrate Arena, hold SSE for one compatibility window if external clients exist, then remove the SSE route/module/docs/generated endpoint. Keep service WS because its canonical/private contract is different.

Net result: one first-party public stream with explicit lag and retention-gap semantics.

9. Do not commit lifecycle states that the policy cannot enter

Fresh-genesis choice

The core oracle implements only Immediate, which transitions Active → Resolved. Yet proposed, challenged, and voided states cross sequencer, API, snapshot, canonical, witness, verifier, and guest schemas.

Evidence: docs and code explicitly label Proposed, Challenged, and Voided as reserved. The current LLM resolver keeps its challenge queue outside core state and eventually submits an immediate signed attestation.

Proposal. If optimistic dispute/void semantics are not a committed next milestone, reduce canonical market status to Active/Resolved and remove proposal/challenge payloads from ResolutionRecord. Reintroduce states with executable transitions, economics, adjudication, and conformance tests. Batch this with another planned fresh genesis.

Net result: the validity schema describes executable protocol truth rather than future possibility. If trust-minimized resolution is imminent, keep it and assign the milestone instead.

10. Generate one audited route registry instead of maintaining four surfaces

Unify metadata

API routes are registered in router builders, repeated in public/owner/service/dev tables, repeated again as exact expected arrays in policy tests, and represented in OpenAPI annotations. This is safety-motivated duplication, but it is still drift-prone.

Evidence: app.rs is 1,103 code lines and tests/route_policy.rs is 1,085. The four route tables contain roughly 80 mounts. Metric labels already use Axum's MatchedPath, so the tables no longer serve that purpose.

Proposal. Define a route declaration that binds handler, method/path, policy class, dev gating, and OpenAPI inclusion once, then emit the grouped routers and an auditable manifest. Preserve independent negative authorization tests and a checked-in policy snapshot; remove hand-copied full arrays.

Net result: adding a route requires one declaration and one deliberate policy review, not synchronized edits across several lists.

11. Define a minimal Compose topology and opt into subsystems

Operational clarity

Base Compose starts eleven services: exchange, history, web, mirror, mock prover, Arena worker/dashboard, VictoriaMetrics, vmalert, Grafana, and node exporter. The deployment-profile environment variable changes settings, but it does not simplify which services run.

Evidence: no Compose profiles: entries were found. A new developer must understand the full product/integration/ops stack before starting the smallest useful exchange.

Proposal. Introduce service profiles such as core (API/history/web), validity (prover/L1 helpers), integrations (Polymarket/Arena), and ops (metrics/alerts). Production selects explicit profiles; local quick start selects core. This is separate from SYBIL_DEPLOYMENT_PROFILE.

Net result: optional products are operationally optional, and the minimal system is visible.

12. Remove dead direct dependencies and centralize versions

Low risk

Manifest repetition and unused direct dependencies create false ownership and make version drift harder to see.

Evidence: static source search found no references for rand/rand_chacha in matching-solver, sha3 in sybil-proof-protocol, or getrandom in sybil-prover. Commonware 2026.5.0 is repeated in sequencer/verifier/zk/escape manifests; direct Reqwest/Axum/redb versions also repeat.

Proposal. Remove the four candidate direct dependencies and verify all targets/features. Move common direct versions to [workspace.dependencies], while retaining per-crate feature selection. Do not attempt to force Commonware's transitive Reqwest/OTel versions to match direct versions without upstream compatibility.

Net result: manifests describe real direct ownership and upgrades have one version locus.

13. Treat the generated Python client as a build artifact or package

Repo hygiene

The generated client dominates Python file and line counts, making the repository look much larger and producing broad mechanical diffs whenever OpenAPI changes.

Evidence: arena/sybil_client/_generated contains 249 tracked Python files and 34,104 lines. The hand-written wrapper imports generated models, so simply deleting the directory would break runtime.

Proposal. Publish the generated SDK as a versioned internal package or generate it deterministically during packaging/CI and verify a schema hash. Keep the ergonomic wrapper and generated-code drift check. Do not hand-prune generated endpoints.

Net result: ordinary reviews focus on authored Python while schema drift remains enforced.

Cross-language and repository hygiene

Python

  • Generated SDK: 249 files / 34,104 lines should be treated as generated output, not manually simplified. Package or regenerate it.
  • Arena runtime: the live bot system is substantial and intentional. Its SQLite read model should remain Python-owned.
  • Two Streamlit products: Arena dashboards are active; the root solver dashboard has schema drift. Do not conflate them.
  • Documentation drift: the Python SDK README says WebSocket-first, while code is SSE-only. Fix behavior first, then docs.

Web

  • The generated TypeScript API schema is 6,608 lines and should remain generated.
  • The largest authored component is arena-view.tsx at 2,008 lines, reinforcing that the Arena product decision has large downstream consequences.
  • Several core UI files are 700–1,400 lines. Split them only by stable state/model boundaries; extracting tiny components without reducing shared state would increase navigation cost.
  • /m-dev/[id] explicitly uses legacy mock data. If the live /dev zone covers the same diagnostics, remove the mocked route in a separate web cleanup.
  • The in-flight computer-use acceptance work was not present in this snapshot. Preserve acceptance scenarios while evaluating the product UI; they are a test surface, not an obvious feature deletion.

Tooling, data, and docs

  • The 824-line justfile has 124 recipes. Remove stale aliases first, then group recipes with imported justfiles or thin domain scripts if navigation remains costly. The proof and deploy recipes encode real operational sequences and should not be deleted merely for length.
  • Benchmark result JSON and frozen market datasets account for much of the 128k JSON lines. Move bulky reproducible outputs to release artifacts/object storage only if clone/review cost is a problem; retain protocol declarations, summaries, hashes, and goldens.
  • frontend/archive is only four documents and clearly marked. It is low-value cleanup compared with runtime ownership.
  • The architecture vault is unusually useful and currently reflects the history and deployment boundaries. Keep it, but remove or mark stale operational instructions at the same time as code paths.

A practical simplification sequence

Do the low-risk truthfulness changes first, then finish already-accepted architectural migrations. Product deletions come after explicit decisions, preferably before the next fresh genesis.

  1. Wave 1 · truthful surface (1–2 days)Remove mock-live, the no-op viz feature, broken sim-eg, and the four candidate dead direct dependencies. Correct Python streaming docs. Decide whether OTLP is temporarily removed or fully scheduled.
  2. Wave 2 · complete existing extractions (3–7 days)Delete sequencer history query caches/persistence while retaining live/rolling state and outbox fact construction. Split the retained-cash production feature from research solvers. Add regression tests for restart, 24-hour aggregates, history delivery, and production solver identity.
  3. Wave 3 · unify interfaces (3–7 days)Migrate Python/Arena to resumable public WS and retire SSE after compatibility review. Replace copied API route inventories with one audited declaration. Add Compose service profiles.
  4. Wave 4 · product boundary decisionsChoose whether the public Arena, native catalog, and LLM resolver are core product. Repair their ownership if kept; remove the full vertical if not. Do not leave default-off code spanning API, store, web, and deployment indefinitely.
  5. Wave 5 · validity cleanup at a fresh genesisIf optimistic/void resolution is not imminent, remove reserved lifecycle variants. Re-evaluate PriceCondition and payoff-vector generality only as a product-strategy decision; batch any canonical changes with the next planned genesis.

Suggested acceptance criteria

  • The production binary can name exactly one matching algorithm, while research commands can still run all comparators.
  • No public history request calls the sequencer actor or reads a sequencer-owned history cache.
  • The API no longer opens a Python-owned database file.
  • Every enabled telemetry exporter has a deployed backend and a documented span/correlation path.
  • Every Cargo feature changes compilation or behavior and has at least one feature-matrix check.
  • Every long-lived sidecar name matches the domain it owns.
  • The minimal local stack is a documented, runnable subset of production—not all optional products at once.
