# In-guest P-256 ECDSA via OpenVM — integration recipe

Shared prerequisite for **SYB-225** (proven key-ops) and **SYB-32** (escape
claims): both verify a secp256r1/P-256 ECDSA signature *inside* the OpenVM guest.
Today the guest verifies **no** signatures (`zk/openvm-guest/openvm.toml` enables
only `rv32i, rv32m, io, sha2`). This note is the confirmed, codex-followable wiring.

## Feasibility: CONFIRMED ✅

OpenVM **v2.0.0-beta.2** — the exact tag already pinned in
`zk/openvm-guest/Cargo.toml` (git `openvm-org/openvm`, resolved commit
`30a13e2`) — supports **secp256r1 / P-256 as a first-class accelerated curve**,
at full parity with secp256k1. Evidence in the resolved cargo checkout:

- `guest-libs/p256/` — a drop-in patch of the crates.io `p256` crate, exporting
  `NistP256`, `P256Point`, `P256Coord`, `P256Scalar`, with `src/ecdsa.rs`.
- `crates/sdk-config/src/openvm_standard.toml` — ships a `# secp256r1 (p256)`
  curve entry (`struct_name = "P256Point"`).
- `crates/circuits/primitives/src/bigint/utils.rs` —
  `secp256r1_coord_prime()` / `secp256r1_scalar_prime()` (`// aka P256`).
- **Copy-paste template:** `guest-libs/p256/tests/programs/examples/ecdsa.rs`
  (no_std P-256 `verify_prehash` against a FIPS 186-4 vector). Also
  `extensions/ecc/tests/programs/openvm_init_ecdsa_recover_p256.rs` (pubkey
  recovery) and `crates/sdk/guest/p256/` (full SDK guest package).

EC ops are **dedicated STARK chips / RV32 precompiles** (`SETUP_EC_ADD_NE_RV32`,
`SETUP_EC_DOUBLE_RV32`, modular-arith chip), NOT soft-Rust execution — so an
in-guest verify is a bounded number of accelerated chip rows, not the tens of
millions of cycles a pure-Rust `p256` verify would burn. This is exactly why the
accelerated extension (not the soft crate) is mandatory. OpenVM benchmarks P-256
ECDSA in its own `benchmarks/guest/kitchen-sink` suite.

### (c) Init + verify API

Init macros are **auto-generated by `cargo openvm build` from `openvm.toml`** into
`openvm_init.rs`; the guest just calls `openvm::init!("openvm_init.rs");` (the
current guest `main.rs` does not call `init!` yet — this is new). The generated
file for P-256 is effectively:
```rust
openvm_algebra_guest::moduli_macros::moduli_init! { "…coord…", "…scalar…" }
openvm_ecc_guest::sw_macros::sw_init! { "P256Point" }
```
Do not hand-write these; let the build generate them.

Verify (from `guest-libs/p256/src/ecdsa.rs`):
- `VerifyingKey = openvm_ecc_guest::ecdsa::VerifyingKey<NistP256>`
- `Signature   = ecdsa_core::Signature<NistP256>`
- key from our stored compressed SEC1: `VerifyingKey::from_sec1_bytes(&pubkey_sec1)`
- verify: `PrehashVerifier::verify_prehash(&prehash, &sig)`  (we supply the
  SHA-256 prehash of the canonical bytes), or the free
  `openvm_ecc_guest::ecdsa::verify_prehashed::<NistP256>(...)`.

This matches our key model exactly: keys are stored as **compressed SEC1 P-256**
(`crates/matching-sequencer/src/crypto.rs`), and key-op/escape canonical bytes are
already prehashed under a domain string — feed that SHA-256 digest to
`verify_prehash`.

## Commitment consequence

Adding the `modular` + `ecc` extensions **changes `app_vm_config` ⇒ moves
`app_vm_commit`** — the first VM-commit move since `0x0026ab66`. Adding the new
guest crate deps also moves `app_exe_commit`. Repin discipline (per SYB-228):
run everything in `~/sybil`, then `just openvm-commit`, refresh the
`OpenVmVerifierAdapter.sol` pin, and `scripts/zk-guest-fingerprint.sh --write`
(no fmt between write and check). Rides the fresh-genesis redeploy window.

## WebAuthn note

The above is **raw-P256 ECDSA** (ratification D0 recommendation). WebAuthn-in-guest
additionally requires parsing the `authenticatorData || SHA-256(clientDataJSON)`
envelope in-circuit before the same `verify_prehash` — deferred unless D0 flips.
The P-256 primitive is identical either way; only the message-construction differs.
