### 1. Economically bound the public API

This is probably the most important launch-adjacent architecture item, and it does not have a good dedicated Linear ticket.

Public account creation currently allows:

- Unlimited permanent accounts.
- Caller-selected demo balances up to $5,000 per account.
- No deposit, invite stock, proof-of-work, or global account cap.
- Every account entering every subsequent full-state witness.
- Append-only fill, account-event, and equity history without a comprehensive byte/age budget.

The audit estimates approximately 405 recurring witness bytes per empty account per block. This makes cheap account creation a permanent proving/DA tax.

Required work includes:

- Separate private-devnet demo funding from public onboarding.
- Require invite stock, faucet allowance, deposit, or another scarce resource.
- Add IP/global account-creation limits and a hard account-stock budget.
- Put row/byte/time retention on append-only history.
- Decide minimum deposit/quarantine economics.

The evidence is in the [DoS audit](/home/anonymous/sybil/design/dos-audit-2026-07-11.md:13). Four findings were fixed, but unlimited accounts, free balances, unpruned history, and quarantine dust remain.

### 5. Production DA, custody, and hostile-operator recovery

The recovery scaffolding is much better than the open tickets imply:

- Full canonical witness exists.
- DA commitments bind proof inputs.
- Witness import can start a replacement chain.
- Custody tooling reconstructs user state.
- Escape claims can be constructed and submitted.
- Backup/import drills exist.

What remains is the difficult external trust layer:

- Independent DA retention outside the operator’s host.
- Encryption rather than plaintext witness publication.
- Defined retention SLO.
- Emergency disclosure/key-release policy.
- Non-operator key holders or another release mechanism.
- A successor-operator appointment/governance mechanism.
- Real deployed verifier adapters.
- Repeated recovery and escape drills against deployed artifacts.

Today recovery works if the bytes survive and are obtainable. It is not yet hostile-operator replacement. The boundary is described in [Operator Replacement](</home/anonymous/sybil/docs/architecture/04-verification/Operator Replacement.md:132>).

Relevant tickets: SYB-32, SYB-80, SYB-116, SYB-120. Several are partly stale because their design or reconstruction acceptance already landed; the remaining work should be reframed as production implementation.

### 6. Complete the validity authorization boundary

The ZK guest verifies state transitions and key registration/revocation authorization. However, ordinary signed orders and cancellations are still checked at API/sequencer admission, not re-proved inside the guest.

Specifically, the guest does not currently receive and prove:

- Ordinary order/cancel signature envelopes.
- The previous durable cross-block replay nonce.

That means a real verifier proves the transition was internally consistent, but still trusts the sequencer’s admission assertion for ordinary user intent. This is documented in [SPEC](</home/anonymous/sybil/docs/SPEC.md:150>) and the [Threat Model](</home/anonymous/sybil/docs/architecture/04-verification/Threat Model.md:11>).

This does not have a clean open Linear issue. It is a substantial witness/schema/guest change and probably implies another fresh genesis.

A related trust boundary is market resolution: one signed feed can settle a market irreversibly. There is no bond, quorum, challenge period, or adjudication mechanism. The exploratory [trust-minimized resolution proposal](/home/anonymous/sybil/design/trust-minimized-resolution.md:62) describes the missing pieces.

## Large scale/code-quality work

### 7. Move proof cost from total state toward activity

The current witness deliberately carries complete pre-state, post-system-state, and post-state snapshots. This was an excellent correctness choice, but it is fundamentally `O(total state)` per block.

At large order/account counts, every block repeatedly proves mostly unchanged state. The likely eventual design is:

- Touched-leaf/opening transition witnesses per block.
- Incremental state proofs.
- A separate periodic full recovery snapshot.
- Hourly or epoch-level proof settlement.
- Explicit retention/availability of epoch recovery data.

This is a genuine architecture redesign and does not have a good Linear implementation ticket. It should wait until real proving measurements show the current wall.

Related later work:

- SYB-105: replace the complete order-book snapshot blob when scale justifies it.
- Retention for fill/event/equity tables.
- A proper prover queue/scheduler rather than file inboxes.
- Historical proof backfill, SYB-87, only after the proof format stabilizes.

I would not do another general sequencer or solver refactor. The large simplification work—actor boundary, verifier consolidation, WAL, persistence, simulation split, canonical hashes, ZK host separation, god-file decomposition—has already landed.

## The actual private-order privacy project

Today `/v1/attestation` is explicitly a development stub. The sequencer does not run inside Nitro, and orders are not encrypted to an enclave.

The complete tranche is:

- SYB-25: Nitro feasibility.
- SYB-26: enclave-bound signing keys.
- SYB-77: client-to-enclave encrypted orders.
- SYB-78: attestation format/verification.
- SYB-82: real attestation endpoint.
- SYB-83: Rust/TypeScript attestation libraries.
- SYB-84: enclave upgrade/PCR governance.
- SYB-85: reproducible enclave images.

This is XL work because it spans clients, cryptography, deployment, recovery, key management, attestation, and upgrades.

It is not required for a basic validium devnet—the current ADR allows the operator to see orders—but it is required if “operator-blind/private orders” remains a product promise.

## My recommended sequence

After the current deploy closeout:

1. Bound public account creation, demo funding, and durable history.
2. Turn CI and guest reproducibility into real required gates.
3. Close SYB-275’s account privacy leak.
4. Add a lean operator-approved market freshness pipeline.
5. Decide the next milestone explicitly:
   - If it is real value: do real proving/L1/DA.
   - If it is product discovery: build the user agent platform.
6. Keep TEE, opportunity markets, and ZK credentials out of the same tranche.

That moves toward a usable devnet without reopening the enormous “make every trust boundary perfect” program.
